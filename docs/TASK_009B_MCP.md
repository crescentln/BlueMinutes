# Task 009B Local MCP Adapter

Status: Completed, pending user acceptance
Date: 2026-07-21
Rollback Git anchor: `b32c7b414af7d12547acf37b1fec0243cfa3e464`
Architecture decision: [ADR-0015](adr/ADR-0015-task-009b-local-mcp-boundary.md)

## Implemented scope

Task 009B adds one independently removable, local-only MCP stdio adapter over
the accepted Task 009A command dispatcher. It adds no new business command,
provider/model route, network surface, credential flow, meeting-content path,
or application UI.

`MeetingBuddyAutomation` owns protocol parsing and tool translation without a
persistence, filesystem, process, or network dependency. `meetingbuddy-mcp` is
the thin executable composition root that opens one already-existing local
workspace and supplies the concrete repository. The application dispatcher
remains the only execution boundary.

## Launch and connection approval

```text
meetingbuddy-mcp --workspace <absolute-path> --approve-read-tools-with-local-audit
```

The exact approval flag is mandatory and explicitly covers bounded local audit
writes. Opening an accepted schema-v8 workspace also creates its verified
rollback backup and migrates the closed audit-origin vocabulary to schema v9.
The executable rejects missing, extra, reordered, relative, root, traversal,
repeated-separator, and symlinked workspace arguments. It neither creates a
workspace nor accepts an authority, role, confirmation, provider, model,
database, or arbitrary path argument.

## MCP surface

The server implements MCP `2025-11-25` initialize/initialized lifecycle,
`ping`, `tools/list`, and `tools/call` over newline-delimited UTF-8 JSON-RPC.
It advertises only `tools` and keeps standard output protocol-only.

| Tool | Input | Application permission | Business-state effect |
| --- | --- | --- | --- |
| `describe_settings` | closed empty object | `read` | audit metadata only |
| `get_command_catalog` | closed empty object | `read` | audit metadata only |
| `get_meeting_policy_status` | exact opaque `meeting_id` UUID | `read` | audit metadata only |
| `get_settings` | closed empty object | `read` | audit metadata only |
| `get_storage_report` | optional `maximum_entries`, 1 through 100000 | `read` | audit metadata only |
| `get_workspace_status` | closed empty object | `read` | audit metadata only |
| `list_activity` | optional `limit`, 1 through 200 | `read` | audit metadata only |

The allowlist is explicit. A future command does not become an MCP tool merely
because its catalog permission is `read`. Settings mutation, rollback,
diagnostics, export, provider/model work, recording, jobs, destructive files,
credentials, policy mutation, arbitrary paths/databases, remote control, and
HTTP remain unexposed.

## Authority, privacy, and failure behavior

- Caller origin is durably `mcp`; actor is `local_mcp`; maximum permission is
  fixed at `read`; client identity and metadata are untrusted and ignored for
  authority.
- Command IDs and replay nonces are generated by the server. Tool arguments
  cannot supply authority, confirmation, recursion, or workspace context.
- Messages and encoded command results are bounded to 1 MiB. Tool calls are
  serialized and limited to 120 per rolling minute by default.
- Unknown tools and malformed input fail before dispatch. Application failures
  return stable safe codes with `isError: true`; raw paths, database/framework
  errors, content, and credentials are not returned.
- Successful results contain the same typed `AutomationCommandExecution` as
  canonical text and structured JSON. Calls write bounded local audit metadata,
  so the tools make no false read-only/idempotent annotation claim.
- No network API, URL session, external process, shell, telemetry, credential,
  Keychain, restricted job directory, meeting-content export, or provider call
  is used by the MCP adapter.

## Provider and fallback disposition

No additional provider was added. Task authorization did not supply a separate
organization/cloud/subscription destination, retention, deployment, account,
quota, credential, or data-policy contract. Adding one would broaden scope and
requires its own explicit approval.

The previously accepted Apple on-device ASR, translation, and Foundation
Models adapters remain available when their local eligibility gates pass.
Manual local transcript/translation/analysis/briefing review remains the
offline fallback. Removing or denying MCP therefore leaves all accepted local
application behavior and local/offline fallback paths intact.

## Persistence and compatibility

Schema v9 migration `009_mcp_audit_origin` truthfully adds `mcp` to the closed
origin vocabulary of `automation_command_records`. It rebuilds only that audit
table and recreates its indexes and immutability triggers. Existing command and
result canonical payloads and SHA-256 values remain byte-for-byte unchanged;
child foreign keys remain valid; no settings or semantic/provider row is
fabricated.

The accepted-v8 migration test verifies a readable source-version-8 online
backup, exact command/result bytes in both the migrated database and backup,
`quick_check`, foreign keys, close/reopen decoding, and a real post-migration
`mcp` audit write. A subsequent injected migration failure rolls back its own
transaction, preserves a valid v9 database, and leaves the exact v8 rollback
backup available.

## Verification coverage

The Task 009B tests cover:

- lifecycle ordering, version negotiation, `ping`, and tools-only capability;
- the exact seven-tool allowlist and closed JSON Schemas;
- hidden write/diagnostic tools and authority injection rejected before the
  executor;
- server-generated identity, typed command substitution, canonical text plus
  `structuredContent`, and safe error redaction;
- malformed and oversized input, incremental EOF framing, rate limiting, and
  adapter failure behavior;
- real disposable-workspace command execution with read authority and durable
  `mcp` attribution;
- fresh and accepted-v1-through-v8 migration compatibility, exact v8 audit
  byte preservation, verified rollback backup, failure injection, and foreign-
  key/integrity checks;
- package/dependency, forbidden import/API, secret/content, executable launch,
  linkage, documentation-link, whitespace, and working-tree checks.

Exact final commands, suite counts, and results are recorded in
`docs/CODEX_EXECUTION_STATE.md`. All protocol and persistence integration uses
synthetic inputs and disposable local workspaces. No real account, provider,
network, credential, meeting content, or user workspace is exercised.

## Removal and residual limits

- Remove the `meetingbuddy-mcp` product/target and adapter source to remove the
  transport; retain v9 to read truthful historical MCP audit origins.
- A full schema downgrade restores the verified pre-migration v8 backup rather
  than altering v9 in place.
- No bundled MCP client configuration or GUI connection workflow is added.
- Real third-party MCP-host interoperability, Xcode GUI/manual release,
  Developer ID signing, notarization, and clean-machine installation remain
  unverified release work.
- HTTP, remote access, providers, accounts, credentials, exports, recording,
  arbitrary files/databases, remote control, and Task 010 remain absent.

No branch, commit, tag, push, release, deployment, user-workspace migration, or
next-task work is included in Task 009B pending explicit user direction.
